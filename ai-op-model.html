AI Operating Model v1.2 · 2026

A compound engineering operating model for a treasury tech team of BSAs, Developers, QA/QE Engineers, and Prod Support. Custom GitHub Copilot agents are embedded at every major handoff. Agents supplement human judgment — they never replace it.

Core Principle
Structured artifact chain per Jira ticket — each agent builds on the last
Tool Philosophy
GitHub Copilot for Dev/QA (VS Code). M365 Copilot for BSA/Prod Support. No direct AI outside enterprise tools.
Regulatory Posture
AI assists, human validates. Full JIRA-ID traceability from requirement to test to runbook.
Quality Gate
No ticket moves to QA without code review. No prod without QA sign-off.

TIERED CHAIN — NOT EVERY CHANGE NEEDS THE FULL WORKFLOW

The chain scales to change complexity. Use judgment — when in doubt, go one tier up. The ai-used tag is required regardless of tier if AI was used at any step.

STANDARD
New feature or significant change
Full chain: BSA agent → Plan agent → Dev agent → Code review → QA → Prod Support. All artifacts in Jira/Xray. Human review at every handoff.
LIGHTWEIGHT
Minor change, enhancement, small bug fix
BSA updates requirements in Jira → Dev agent or manual dev → QA sign-off. Plan agent optional. Prod Support as needed.
EXPEDITED
Hotfix, config change, urgent prod fix
Manual dev → peer review → QA sign-off. AI is optional. If AI was used, tag it. Post-hoc chain review required within one sprint.
REQUIREMENTS CHANGE PROPAGATION RULE
If requirements or BSA test cases are updated after the plan agent has already run, the plan agent must be re-run before dev continues. No exceptions. This prevents artifact drift between what was specified and what was built.

FEEDBACK LOOPS — WHERE THE CHAIN REVERSES

Defects caught here are cheaper than defects caught in prod. Each loop has a named artifact so nothing falls through informally.

Post-Incident→All
Chain Trace
Incident mapped back to artifact chain. Artifacts updated and Xray regression suite updated accordingly.

BSA and QA own distinct test suites with clear boundaries. JIRA-ID naming throughout. Xray (JTMF) is the system of record. The dev agent emits a results manifest that syncs automatically.

ID FORMAT

QA TEST SUITE
System confidence — does the feature work in context of everything else?
ID FORMAT
JIRA-1234-QA01, QA02…
CASE TYPES
·Integration — feature in context of other systems (e.g. Murex → downstream)
·Regression — impact on existing functionality
·Full journey — end-to-end user/system flows
INFORMED BY
→QA agent gap analysis on BSA testplan + requirements
→QA agent regression impact analysis against Xray suite
→Xray QA suite + canonical regression suite
TEST RESULTS MANIFEST — CLOSING THE 10 → 40 TEST GAP

The dev agent runs 10 BSA test cases but also generates ~30 unit/component tests. Without a manifest those extra runs are invisible. The manifest makes every run accountable and pushes outcomes back to Xray.

MANIFEST STRUCTURE — PER BSA CASE
id: JIRA-1234-TC01
status: PASS | FAIL | MANUAL_REQUIRED | SKIPPED
automation: automated | manual | tbd
reason: # if not run — why
type: functional | integration | regression
owner: bsa | qa
STATUS DEFINITIONS
PASSAgent ran and verified
FAILAgent ran, assertion failed
MANUAL_REQUIREDCannot automate — needs human QA
SKIPPEDAgent couldn't test, reason logged
Unit/component tests (~30 extras) grouped separately — visible to devs but not pushed to Xray individually.
XRAY (JTMF) — REGRESSION SUITE VERSIONING

Xray is the system of record. The canonical regression suite always holds the current version — superseded cases are archived, not deleted. QA agent identifies which cases a new requirement touches.

HOW VERSIONING WORKS
·JIRA-1234-TC01 tests addition of two values
·JIRA-5678 changes logic to subtraction
·TC01 archived → JIRA-5678-TC01 is the replacement
·Regression suite holds current canonical case only
·QA agent flags which Xray cases a new req touches
·Traceability bidirectional: requirement ↔ test case
MANIFEST → XRAY PUSH FLOW
1.Dev agent emits JIRA-####-test-results-manifest.json
2.Wrapper script calls Xray REST API (Xray JSON format)
3.PASS/FAIL results land against correct TC-IDs in Xray
4.MANUAL_REQUIRED cases surface as TODO for human QA
5.BSA and QA suites remain in separate Xray components

OPERATING RULES — NON-NEGOTIABLE
ai-used TAG — MANDATORY
The ai-used Jira field must be completed on every ticket close, regardless of tier. This is a required workflow field — not a label or convention. Jira admin to enforce. Enables ROI measurement and outcome tracking.
CHANGE PROPAGATION RULE
If requirements or BSA test cases change after the plan agent has run, the plan agent must be re-run before dev continues. No exceptions. Prevents artifact drift between specification and implementation.
REFERENCE LIB APPROVAL
No reference library file (architecture.md, security-controls.md) may be consumed by an agent without a human-approved merge. CODEOWNERS enforces this — no exceptions.
HUMAN ACCOUNTABILITY
Agents are tools. Every role remains accountable for the artifacts they own. BSA owns requirements and BSA test cases. QA owns QA test cases and sign-off. Dev owns code. AI output does not transfer accountability.
